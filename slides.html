Block 3 · Demo + Exercise

Prompt Structuring

0:20 – 0:30

Context engineering,
not just prompting

You're curating the smallest set of high-signal tokens to maximize the model's chance of success.

  • Context rot — accuracy drops as token count grows
  • Attention budget — like human working memory, it's finite
  • The quality of your input determines the quality of the output

Source: Anthropic, "Effective Context Engineering for AI Agents"

Vague vs. structured

Vague

"Add a search feature"

Structured

"Add a search endpoint to the API: accepts a query param q, searches the title and description fields in the contacts table, returns paginated JSON matching the existing list endpoint format. Add tests."

Let's run both live and compare the output. 

# 1. Run the vague prompt → see what happens
# 2. Reset:  git checkout . && git clean -fd
# 3. Run the structured prompt → compare

The DECOMPOSE pattern

  • Define the outcome
  • Existing patterns to follow
  • Constraints and edge cases
  • Output format expected
  • Measure of done
2–3×
quality improvement when you give
the agent verification criteria
Exercise — 6 minutes

Add a "favorite contacts" feature to the CRM:

  1. Write a structured prompt using DECOMPOSE before sending
  2. Run it through Claude Code
  3. Compare output with a neighbor — whose prompt produced more complete code?

Stretch: Refine the prompt based on what Claude got wrong, re-run, see the improvement.

Block 4 · Demo + Exercise

Task Orchestration & Iteration

0:30 – 0:45

The orchestration loop

Explore → Plan → Implement → Commit

Official Claude Code best practice. Not every task needs every step.

Demo: building a feature end-to-end

Adding email notifications when a deal moves to "Closed Won":

  1. Explore — "Read the deal_service and understand how stage transitions work."
  2. Plan — "Create an implementation plan for email notifications. Don't write code yet."
  3. Implement — "Go ahead. Run tests after each major change."
  4. Debug — "I got this error: [paste]. Fix it."
  5. Verify — "Run the tests and show me the results."
[ Screenshot: Plan mode output → implementation → test results ]

Session management

CommandWhat it does
/clearReset context between unrelated tasks
/compactSummarize conversation to free context
EscStop Claude mid-action (context preserved)
Esc + EscRewind to a previous checkpoint
--resumePick up a previous conversation
/btwSide question — doesn't enter history

Anti-patterns to avoid

  • The kitchen sink session
    Mixing unrelated tasks in one session. Fix: /clear between tasks.
  • Correcting over and over
    After two failed corrections, /clear and write a better prompt.
  • The trust-then-verify gap
    Always include verification criteria. You are the only feedback loop otherwise.

"You're a tech lead doing code review in real-time, not a developer typing."

Exercise — 10 minutes

Build a feature using the orchestration loop:

  1. Choose a task: rate limiting, CSV export, or admin role
  2. Explore → Plan → Implement → Verify
  3. Feed errors back. Get to working code through the loop, not one shot.

Stretch: Use /compact mid-session, or create a slash command for a repeated step.

Block 5 · Demo + Exercise

Multi-Agent Patterns

0:45 – 1:00

Pattern 1: Writer / Reviewer

Tab 1 — Writer

> Implement a rate limiter
  for our API endpoints

Tab 2 — Reviewer

> Review the rate limiter in
  @app/routers/deals.py.
  Look for edge cases, race
  conditions, and consistency.

Feed review back to Tab 1. A fresh context produces unbiased review — Claude won't defend code it just wrote.

Pattern 2: Subagents

Specialized agents that run in their own context window.

> Use a subagent to research how the existing codebase handles
  file uploads. Report back the patterns used.

Built-in subagents

  • Explore — Haiku, read-only, fast
  • Plan — research for planning
  • General-purpose — full tools

Why subagents?

  • Keeps main session context clean
  • Limit tool access per agent
  • Route to cheaper/faster models

Pattern 3: Agent Teams

Multiple Claude Code instances coordinating via shared task lists and messaging.

  • Team lead — decomposes work, spawns teammates, synthesizes results
  • Teammates — independent context, own task queues
  • Shared task list — pending / in-progress / completed
  • Mailbox — inter-agent messaging
